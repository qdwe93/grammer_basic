# 02. 데이터 모델

이 문서는 퀴즈 한 문제가 어떤 형태의 데이터인지, 라벨 종류는 무엇인지, 채점은 어떻게 하는지를 정의한다.
저장된 JSON 데이터는 **이 스키마를 따라야 한다.**

## 1. 워크북 구조 분석 (출처)

`resource_docs/`의 워크북을 분석한 결과, 한 문장은 다음 행(row)들로 구성된다.

```
번호(index)        : 1   2       3     4
단어(word)         : A   country mouse lived ...
품사(pos)          : 단어마다 1개   (명사/대명사/동사/형용사/부사/전치사/접속사)
문장성분(component): 구(phrase)마다 1개 (주어/서술어/목적어/보어/수식어) — 여러 단어에 걸침
해석(chunk)        : 문장성분 구 단위 한글 뜻
해석(문장)         : 문장 전체를 한글 어순으로 재배열한 해석
```

→ 핵심: **품사는 단어(token) 단위, 문장성분은 구(phrase = 연속된 토큰 묶음) 단위.** 데이터 모델은 이 둘을 분리해 표현한다.

## 2. 라벨 정의 (enum)

### 2.1 품사 (Part of Speech) — `pos`
`design.md`의 7개 역할과 1:1 대응한다.

| key | 한글 | 색상 토큰(틴트/잉크) |
|---|---|---|
| `noun` | 명사 | `pos-noun` |
| `pronoun` | 대명사 | `pos-pronoun` |
| `verb` | 동사 | `pos-verb` |
| `adjective` | 형용사 | `pos-adjective` |
| `adverb` | 부사 | `pos-adverb` |
| `preposition` | 전치사 | `pos-preposition` |
| `conjunction` | 접속사 | `pos-conjunction` |

> **관사(a/the) 처리(중요)**: 원본 워크북·디자인 팔레트에는 별도 "관사" 카테고리가 없다. 본 앱에서는 관사를 **`adjective`(형용사/한정사)** 로 분류한다. 시드 데이터가 이 규칙을 따른다. (만약 추후 "관사"를 독립 품사로 분리하려면, 팔레트에 8번째 색을 추가하고 enum·범례를 함께 갱신해야 한다 — 현재는 7종 고정.)

### 2.2 문장성분 (Sentence Component) — `component`
5개 family. 목적어는 직접/간접으로, 보어는 주격보어를 포함하되 같은 family 색을 쓰고 칩 라벨로 구분한다.

| key | 한글(칩 라벨) | family 색상 토큰 |
|---|---|---|
| `subject` | 주어 | `comp-subject` |
| `predicate` | 서술어 | `comp-predicate` |
| `object` | 목적어 | `comp-object` |
| `object_direct` | 직접목적어 | `comp-object` (같은 색, 칩만 다름) |
| `object_indirect` | 간접목적어 | `comp-object` (같은 색, 칩만 다름) |
| `complement` | 보어 | `comp-complement` |
| `modifier` | 수식어 | `comp-modifier` |

> UI 선택기에서는 보통 `주어 / 서술어 / 목적어(직접·간접) / 보어 / 수식어`로 노출하고, 정답이 `object_direct`/`object_indirect`인 경우 채점은 family(`object`) 일치 또는 세부 일치 중 무엇을 정답으로 볼지 [채점 규칙](#5-채점점수-규칙)에서 정한다.

## 3. JSON 스키마

### 3.1 최상위 — 문제집(Deck)

```jsonc
{
  "id": "town-mouse",
  "title": "The Town Mouse and the Country Mouse",
  "language": "en",
  "source": "seed",
  "questions": [ /* Question[] */ ]
}
```

### 3.2 문제(Question) = 문장 1개

```jsonc
{
  "id": "q1",
  "sentence": "A country mouse lived in a small hole near a farm.",
  "tokens": [ /* Token[] */ ],          // 단어 단위 (품사 정답 포함)
  "phrases": [ /* Phrase[] */ ],        // 구 단위 (문장성분 정답 + chunk 해석)
  "translation": "농장 근처의 작은 구멍 속에서 시골 쥐 한 마리가 살았다."  // 해석(문장)
}
```

### 3.3 토큰(Token) = 단어 1개

```jsonc
{
  "index": 1,            // 워크북 번호 (1부터)
  "text": "A",
  "pos": "adjective"     // 정답 품사 (2.1의 key)
}
```
- 구두점(쉼표·마침표·따옴표)은 직전 단어 `text`에 붙이거나 별도 비채점 토큰으로 둘 수 있다. 권장: **단어에 붙이지 말고, 채점 대상이 아닌 토큰에는 `pos: null`** 을 주어 선택기를 렌더하지 않는다.

### 3.4 구(Phrase) = 문장성분 1개

```jsonc
{
  "id": "p1",
  "tokenIndexes": [1, 2, 3],     // 이 구에 속한 토큰 번호들 (연속)
  "component": "subject",        // 정답 문장성분 (2.2의 key)
  "chunkGloss": "시골 쥐 한 마리가"   // 해석(chunk) — 이 구의 한글 뜻
}
```
- `phrases`는 문장의 모든 채점 대상 토큰을 빠짐없이 덮어야 한다(겹치지 않게).
- 줄바꿈으로 브래킷이 끊기면 정렬된 두 세그먼트로 렌더(`design.md` Responsive).

## 4. 시드 데이터 예시 (생쥐 이야기 도입부)

실제 시드 파일은 [`seed/town_mouse.json`](seed/town_mouse.json)에 저장되어 있다. 아래는 첫 두 문장 예시.

```jsonc
{
  "id": "q1",
  "sentence": "A country mouse lived in a small hole near a farm.",
  "tokens": [
    { "index": 1,  "text": "A",       "pos": "adjective" },
    { "index": 2,  "text": "country", "pos": "adjective" },
    { "index": 3,  "text": "mouse",   "pos": "noun" },
    { "index": 4,  "text": "lived",   "pos": "verb" },
    { "index": 5,  "text": "in",      "pos": "preposition" },
    { "index": 6,  "text": "a",       "pos": "adjective" },
    { "index": 7,  "text": "small",   "pos": "adjective" },
    { "index": 8,  "text": "hole",    "pos": "noun" },
    { "index": 9,  "text": "near",    "pos": "preposition" },
    { "index": 10, "text": "a",       "pos": "adjective" },
    { "index": 11, "text": "farm.",   "pos": "noun" }
  ],
  "phrases": [
    { "id": "p1", "tokenIndexes": [1,2,3],     "component": "subject",   "chunkGloss": "시골 쥐 한 마리가" },
    { "id": "p2", "tokenIndexes": [4],         "component": "predicate", "chunkGloss": "살았다" },
    { "id": "p3", "tokenIndexes": [5,6,7,8],   "component": "modifier",  "chunkGloss": "작은 구멍 속에서" },
    { "id": "p4", "tokenIndexes": [9,10,11],   "component": "modifier",  "chunkGloss": "농장 근처의" }
  ],
  "translation": "농장 근처의 작은 구멍 속에서 시골 쥐 한 마리가 살았다."
}
```

> 참고: 위 라벨링은 초급 문법 기준의 한 해석안이다. 실제 워크북 정답과 미세하게 다를 수 있으므로, 시드 데이터를 늘릴 때는 `resource_docs/워크북 정답_*.pdf`를 기준 진실로 삼아 교정한다.

## 5. 채점·점수 규칙

### 5.1 정오 판정
- 토큰 품사: 학습자 선택 `pos` === 정답 `pos`.
- 구 문장성분: 학습자 선택 `component` === 정답 `component`.
  - 목적어 세부(직접/간접): 기본은 **family 일치(`object`)면 정답** 으로 관대하게 채점하고, 세부 구분은 정답 공개 시 칩 라벨로 보여준다. (난이도 옵션으로 "세부까지 일치" 모드를 둘 수 있음 — MVP는 family 일치.)

### 5.2 재시도 흐름
- 채점 시 틀린 항목 집합만 표시. 정답 비공개.
- 학습자가 고쳐 재채점. 전부 정답이 될 때까지 반복.
- 각 항목에 대해 **"한 번이라도 틀린 적 있음(wasWrong)"** 플래그를 기록 → 공개 시 `was-wrong` 마커 + 점수 산정에 사용.

### 5.3 점수(정답률)
- 권장 정의: 문제(문장)별로 **모든 항목을 한 번의 채점에서 틀림 없이 맞히면 1점(perfect)**, 한 번이라도 틀렸다 고쳐 맞히면 부분점(예: 0.5) 또는 0점.
- 최종 정답률 = (perfect 문제 수 등 가중합) / (전체 문제 수).
- 이 정답률로 단계 4의 보상 티어를 결정.
- 구체 가중치는 구현 시 상수로 분리(`SCORING` 설정)하여 쉽게 조정 가능하게 한다.

## 6. 검증 규칙
데이터를 사용하기 전 다음을 검증한다(Zod).
1. 모든 `tokens[].index`가 1부터 연속.
2. 채점 대상 토큰(`pos != null`)이 정확히 하나 이상의 `phrase`에 속함(누락·중복 없음).
3. 모든 `pos`가 2.1 enum, 모든 `component`가 2.2 enum.
4. `phrases[].tokenIndexes`가 연속 구간.
5. `translation`, 각 `chunkGloss` 비어있지 않음.
- 검증 실패 시: 콘솔에 오류를 남기고 로드를 중단한다.
